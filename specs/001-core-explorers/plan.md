# Implementation Plan: Phase 1 Core Explorers

**Branch**: `001-core-explorers` | **Date**: 2026-03-06 | **Spec**: `/specs/001-core-explorers/spec.md`
**Input**: Feature specification from `/specs/001-core-explorers/spec.md`

## Summary

Implement Phase 1 core artifact exploration by adding four specialized explorers (TypeScript, Python, JSON, Markdown) plus a strengthened fallback path, all behind existing `ExplorerPort` and `ExplorerRegistryPort` boundaries. Update resolver behavior to satisfy weighted deterministic selection, enforce standardized exploration metadata (including score/confidence and truncation), support structured failure payloads for known unreadable/malformed inputs, and preserve operation-level exceptions for unexpected internal faults. Ensure outputs remain token-budget compliant (including stratified sampling for large artifacts) and verified through deterministic unit/integration tests.

## Technical Context

**Language/Version**: TypeScript 5.x (strict), Node.js >=22.0.0
**Primary Dependencies**: `@ledgermind/application`, `@ledgermind/domain`, `@ledgermind/adapters`, Vitest 3.x, ESLint + `eslint-plugin-boundaries`, Turborepo
**Storage**: Existing artifact persistence (`ArtifactStorePort`) over in-memory and PostgreSQL adapters; no new storage backend
**Testing**: Vitest unit + integration + conformance-style explorer tests; deterministic repeated-run selection tests
**Target Platform**: Node.js library runtime (SDK composition root + adapters)
**Project Type**: TypeScript monorepo library (`packages/application`, `packages/adapters`, `packages/sdk`)
**Performance Goals**: 100% token-limit compliance for exploration outputs, 100% deterministic explorer selection for repeated identical inputs, non-empty exploration payloads across all five Phase 1 categories
**Constraints**: Weighted scoring across extension/content-type/sniffing with deterministic tie-break; exploration is read-only; known unreadable/malformed cases return structured failures; unexpected internal faults raise typed operation errors; coarse failure classes only (`unsupported-readable`, `unsupported-unreadable`, `malformed-structured`)
**Scale/Scope**: Phase 1 only — five core explorers (TS, Python, JSON, Markdown, Fallback), registry resolution rules, and required metadata/failure contract enforcement

## Phase Delivery Strategy

### Phase 0 — Research

1. Define weighted scoring model using all required signals (extension, mime/content-type, sniffing) and deterministic tie-break semantics.
2. Define standardized metadata contract that satisfies FR-009a/FR-009b for success and structured-failure outputs.
3. Define structured-failure model (known unreadable/malformed returns payload, unexpected internal faults throw).
4. Define stratified sampling baseline for large artifacts and truncation disclosure rules.

### Phase 1 — Design & Contracts

1. Specify explorer-specific output contracts for TypeScript, Python, JSON, Markdown, and fallback.
2. Specify resolver decision model and candidate scoring/tie-break behavior.
3. Specify failure classification mapping and actionable guidance requirements.
4. Define test contract for deterministic selection, token-budget enforcement, and edge cases.

### Phase 2 Preparation (for `/speckit.tasks`)

1. Implement specialized explorers and shared helper utilities.
2. Update registry/default wiring and weighted resolver behavior.
3. Update fallback and use-case orchestration to enforce metadata/failure contracts.
4. Add/extend tests for explorer outputs, resolver determinism, structured failures, and sampling/truncation.
5. Run workspace quality gates (`build`, `typecheck`, `lint`, `test`).
