# Implementation Plan: PostgreSQL Adapter for Phase 1 Core Engine

**Branch**: `001-postgres-adapter` | **Date**: 2026-03-02 | **Spec**: `/specs/001-postgres-adapter/spec.md`
**Input**: Feature specification from `/specs/001-postgres-adapter/spec.md`

## Summary

Implement the Phase 1 PostgreSQL persistence surface for LedgerMind by delivering concrete PostgreSQL adapters, SQL schema/migrations, and PostgreSQL conformance validation for all persistence-critical use-case paths: conversation lifecycle, immutable ledger append/read/search, context projection with optimistic concurrency, summary DAG persistence/expansion/integrity checks, artifact storage, and atomic transaction orchestration through `UnitOfWorkPort`.

The implementation must preserve Clean Architecture boundaries, deterministic behavior, and typed error surfaces while adding bounded internal retry behavior for transient PostgreSQL failures per feature clarification FR-017.

## Technical Context

**Language/Version**: TypeScript 5.x (strict), Node.js >=22.0.0
**Primary Dependencies**: `@ledgermind/domain`, `@ledgermind/application`, `pg`, `node-pg-migrate`, Vitest 3.x, ESLint + `eslint-plugin-boundaries`, pnpm/Turborepo
**Storage**: PostgreSQL (Phase 1 persistence backend in scope)
**Testing**: Vitest unit/integration + PostgreSQL harness tests + conformance/golden/regression suites (PostgreSQL scope for this feature)
**Target Platform**: Cross-platform Node.js library runtime for SDK and adapters
**Project Type**: TypeScript monorepo library (`packages/*`)
**Performance Goals**:
- >=95% of append/context retrieval/summary expansion operations complete in <=1s under defined validation scale (SC-004)
- No partial writes on failed multi-record operations (SC-003 + FR-011)
- All integrity checks pass for required golden/conformance scenarios (SC-005)
- Validation scale support up to 10,000 events/conversation and 100 concurrent conversations (SC-008)
**Constraints**:
- Strict dependency rule: `domain <- application <- adapters <- infrastructure <- sdk`
- Immutable ledger append-only semantics
- Explicit optimistic concurrency via `expectedVersion` + stale conflict errors
- Typed retryable vs non-retryable persistence error outcomes
- PostgreSQL-only conformance in this feature (cross-backend parity deferred)
- Bounded internal retry on transient persistence failures
**Scale/Scope**:
- Deliver PostgreSQL adapters and migrations for all Phase 1 persistence ports
- Validate FR-001..FR-019 and SC-001..SC-008 for PostgreSQL scope
- Exclude multi-tenant/server mode and SQLite parity in this feature
