# Implementation Plan: SDK Entrypoint and Vercel Memory Adapter

**Branch**: `001-sdk-vercel-adapter` | **Date**: 2026-03-07 | **Spec**: `/specs/001-sdk-vercel-adapter/spec.md`
**Input**: Feature specification from `/specs/001-sdk-vercel-adapter/spec.md`

## Summary

Deliver the two requested Phase 1 capabilities by preserving the existing SDK composition root and extending it with named presets, then adding a Vercel AI SDK adapter that exposes memory tools (`grep`, `describe`, `expand`) through stable runtime-callable definitions. The adapter must enforce restricted-operation authorization through existing application policies, return one consistent structured success/error envelope, preserve follow-up references for chained memory workflows, and include automated tests that cover all defined negative-path scenarios.

## Technical Context

**Language/Version**: TypeScript 5.x (strict), Node.js >=22.0.0
**Primary Dependencies**: `@ledgermind/domain`, `@ledgermind/application`, `@ledgermind/adapters`, `@ledgermind/infrastructure`, Vitest 3.x, ESLint + `eslint-plugin-boundaries`, Turborepo, Vercel AI SDK package (`ai`) for adapter binding (adapter layer only)
**Storage**: Existing in-memory and PostgreSQL engine creation paths (no new storage backend)
**Testing**: Vitest unit/integration tests across SDK and adapter boundaries, with explicit negative-path contract tests
**Target Platform**: Node.js library runtime (SDK + adapter packages)
**Project Type**: TypeScript monorepo library (`packages/sdk`, `packages/adapters`, `packages/application`)
**Performance Goals**: Deterministic tool envelope shape for identical outcomes, 100% automated coverage of defined negative-path scenarios, no materialized-context over-budget outcomes in covered tests
**Constraints**: Preserve Clean Architecture dependency rule; keep framework types out of domain/application; enforce runtime authorization for restricted expansion; use one canonical structured error envelope for all adapter tool failures; maintain stable SDK-facing engine contract
**Scale/Scope**: Feature scope limited to SDK public entrypoint enhancements (generic + presets) and one Vercel adapter integration in Phase 1

## Phase Delivery Strategy

### Phase 0 вЂ” Research

1. Confirm current SDK composition and identify minimal preset extension strategy.
2. Confirm application-level retrieval and authorization semantics for adapter mapping.
3. Define canonical success/error envelope and typed error mapping approach.
4. Define negative-path test matrix required to satisfy FR-012.

### Phase 1 вЂ” Design & Contracts

1. Define data model for initialization request, tool execution context, and result envelope.
2. Define SDK + Vercel adapter contract with runtime authorization rules.
3. Define quickstart validation flow and quality-gate checkpoints.

### Phase 2 Preparation (for `/speckit.tasks`)

1. Implement SDK presets and public exports while preserving generic factory behavior.
2. Implement Vercel tool adapter and envelope/error-mapping utilities.
3. Add adapter tests for success, restricted denial, invalid references, and internal-failure mapping.
4. Extend SDK tests for generic/preset parity and invalid initialization paths.
5. Run workspace quality gates (`build`, `typecheck`, `lint`, `test`).
