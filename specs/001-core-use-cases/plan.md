# Implementation Plan: Phase 1 Core Use Cases

**Branch**: `001-core-use-cases` | **Date**: 2026-03-01 | **Spec**: `/specs/001-core-use-cases/spec.md`
**Input**: Feature specification from `/specs/001-core-use-cases/spec.md`

## Summary

Implement the Phase 1 core use-case layer for LedgerMind by building on the already-implemented `packages/domain` model and the existing `packages/application/src/ports/**` contract surface. The feature delivers production-ready orchestration for append, context materialization, compaction, retrieval tools, artifact flows, and integrity reporting, with deterministic behavior and typed failures. Validation scope is explicitly limited to in-memory and PostgreSQL adapters and must prove budget safety, compaction convergence, retrieval recoverability, and artifact ID propagation.

## Technical Context

**Language/Version**: TypeScript 5.x (strict), Node.js >=22.0.0
**Primary Dependencies**: `@ledgermind/domain`, `@ledgermind/application` contracts, Vitest 3.x, ESLint + `eslint-plugin-boundaries`, TypeScript project references, Turborepo, `pg` + SQL migrations for PostgreSQL adapter
**Storage**: In-memory + PostgreSQL (Phase 1 validation scope only; SQLite out of scope)
**Testing**: Vitest unit/integration/conformance/golden/regression suites with deterministic test stubs (`DeterministicSummarizer`, `SimpleTokenizer`) where required
**Target Platform**: Cross-platform Node.js library consumers (SDK + framework adapters in later phases)
**Project Type**: TypeScript monorepo library (`packages/*`)
**Performance Goals**: (1) 100% of materialization responses within computed budget or typed budget/convergence error, (2) deterministic compaction termination within configured `maxRounds`, (3) retrieval workflow success >= 90% for search -> describe -> expand validation
**Constraints**: Strict Clean Architecture dependency rule, deterministic ID canonicalization semantics, optimistic context versioning, no partial state corruption on failures, no operator recursion in this phase, no framework/runtime types in domain/application layers
**Scale/Scope**: Phase 1 core engine use cases only (append/materialize/compaction/retrieval/artifact/integrity) with in-memory and PostgreSQL parity checks

2. Define deterministic test harness assumptions (stubs, fixtures, replay guarantees).
3. Produce explicit risk register and mitigation hooks used to shape tasks.

### Phase 1 вЂ” Design & Contracts Finalization

1. Define use-case orchestration boundaries and transaction responsibilities.
2. Map all FR-001..FR-016 to concrete method contracts and error surfaces.
3. Define conformance parity requirements for in-memory and PostgreSQL.
4. Define quickstart validation paths matching SC-001..SC-007.

### Execution Order (for tasks generation)

1. Implement in-memory adapters + deterministic stubs.
2. Implement P1 use-cases (`append`, `runCompaction`, `materializeContext`) and tests.
3. Implement retrieval/artifact/integrity use-cases and tests.
4. Implement PostgreSQL adapters + migrations + conformance parity.
5. Implement SDK composition root and end-to-end validation.
